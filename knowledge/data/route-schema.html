Route Schema — the tagging target

The canonical record for one multi-pitch route. When Phase 1/2 finds a climb on the web, this is the shape we tag it into and the set of facts we need to describe it.

Provenance: learned from the mature multi-pitch.com project (/dev/multi-pitch) — a battle-tested data model behind ~40 fully-described routes. This is the real "compendium of terms" that guidebook-quality descriptions are built from. Controlled vocabularies live in taxonomy.md; the normalized difficulty ladder in grade-conversion.md.

Why this matters for the engine

The vision's Taxonomy Engine (Phase 2) takes messy guidebook/UKC/MountainProject prose and must emit a clean, complete route record. This schema is the output contract. Every field is either a closed enum (validate hard), a number, a boolean flag, or prose (generated to the style rules below). A found climb is "fully tagged" when these fields are populated with provenance.

Describing the route — content style rules

Learned from multi-pitch.com's content rules. When the Taxonomy Engine writes the prose fields, follow these so generated descriptions read like the curated site:

  1. Voice: never "I" or "we" — say "the climber".
  2. Plain language: avoid unqualified mountain jargon, or qualify it. Not *"follow the

col to the dihedral before the arête" → "follow the gully between the rocks until you see the corner before the exposed edge on the left."*

  1. Informal is fine.
  2. Facts earn their place: include geology, first/notable ascent, and history — these

are what make a description worth reading.

  1. Pitch format: Pitch N – <length>m <grade> then the moves, one block per pitch.
  2. Reference-link prefixes (for scannability): Video:, Travel:, Article:,

Info:, Tides:. e.g. Video: Chris Bonington Climbs the Old Man of Hoy.

  1. Approach block always splits into Approach: and Descent:.

Minimal vs full tagging

A found climb is usable once it has: routeName, cliff, geoLocation, originalGrade+gradeSys, length, pitches, rock, and a mapped dataGrade. Everything else (flags, prose, media, references) enriches it toward guidebook quality.

Rules for the parser (Phase 2)

  1. Keep originalGrade verbatim and always store gradeSys — never a bare number.
  2. Map to dataGrade via grade-conversion.md for cross-venue

sorting.

  1. Validate enums (rock, incline, face, gradeSys) against taxonomy.md;

repair or quarantine off-dictionary values.

  1. Hazard flags are safety-critical — only set loose/tidal/seepage from explicit

source evidence; carry the source span.

  1. Prose to the style rules above, with provenance + confidence.
  2. status: draft anything not yet human-verified — only publish surfaces.
